# 企业级智能客服与知识库平台设计文档

> 面向私有化部署场景，支持 Web、独立 H5、嵌入 H5、iOS、Android，前端采用 React + shadcn/ui + TanStack Router，移动端采用 Flutter，后端采用 Python。

## 1. 项目目标

构建一套面向企业私有化部署的智能客服与知识库平台，覆盖访客接入、会话中心、坐席工作台、知识库、AI 客服、运营分析、审计合规和多端交付能力。

平台需要同时支持以下产品形态：

- Web 坐席工作台与后台管理端
- 独立 H5 客服页
- 可嵌入第三方 App 或小程序容器的 H5 客服页
- Flutter iOS/Android 客户端

## 2. 设计原则

- 私有化优先：部署、模型接入、存储、审计都要支持内网落地
- 平台优先：先建设会话、消息、知识、权限、审计底座，再叠加高级能力
- AI 可控：AI 客服必须基于知识、可追溯、可转人工、可审计
- 统一会话真相源：AI 与人工共用同一套会话、消息、客户数据和事件日志
- 多端一致，形态分层：Web/H5/Flutter 能力统一，交互按终端差异化
- 发布与回滚可控：知识、AI 策略、渠道配置必须具备发布流

## 4. 总体架构

采用“平台主服务 + 实时消息中心 + AI 服务中心”的混合架构。

### 4.1 前端层

- `admin-web`
  - React
  - TanStack Router
  - TanStack Query
  - shadcn/ui
  - Tailwind CSS
- `customer-h5`
  - React
  - TanStack Router
  - TanStack Query
  - 支持独立访问和嵌入模式
- `mobile-flutter`
  - Flutter
  - 支持完整原生体验
  - 保留加载 H5 会话页的能力

### 4.2 后端层

- `platform-api`
  - FastAPI
  - 身份权限、渠道配置、客户中心、会话管理、知识管理、报表配置
- `message-gateway`
  - WebSocket
  - 在线状态、消息推送、未读数、回执
- `ai-service`
  - FastAPI
  - RAG 编排、模型网关、AI 决策、转人工策略
- `worker-jobs`
  - 异步任务
  - 文档解析、切片、索引、报表聚合、评测任务

### 4.3 数据层

- PostgreSQL：事务数据真相源
- Redis：缓存、短期上下文、限流、任务状态
- OpenSearch：知识检索、消息搜索、混合检索、分析
- MinIO：附件、知识原文件、导入导出文件

### 4.4 模型层

- 主模型：Qwen 或 DeepSeek
- 推理后端：vLLM 为主，Ollama 为辅
- 第三方模型接入：通过统一模型网关适配

### 5.5 customer-service

职责：

- 客户资料

### 5.6 knowledge-service

职责：

- FAQ、文档知识、流程知识、快捷语
- 分类、标签、版本、审核、发布

### 5.7 ai-service

职责：

- 模型网关
- 检索与问答编排
- 意图识别
- 低置信度转人工

### 5.8 analytics-service

职责：

- 客服报表
- 满意度与留言报表
- 机器人命中、转人工、无答案分析

## 6. AI 客服架构

AI 客服不是自由聊天模型，而是受控的客服问答系统。

### 6.1 在线链路

```text
客户端消息
-> message-service
-> conversation-service
-> policy-check
-> retrieval
-> model-gateway
-> result-judge
-> answer / clarify / handoff / reject
-> message-service 推送结果
-> trace/audit
```

### 6.2 决策模式

- `bot_only`
- `ai_first`
- `human_only`
- `human_takeover`

### 6.3 模型网关

统一屏蔽模型差异，支持：

- Qwen
- DeepSeek
- vLLM
- Ollama
- 第三方 OpenAI 风格接口

### 6.4 AI 安全策略

- 敏感词与高风险话题前置拦截
- 身份与权限校验
- 知识引用强约束
- 低置信度转人工
- 全链路审计

## 7. 知识库设计

知识库的重点是治理、元数据、发布和评测，不只是切片。

### 7.1 知识建模

知识类型：

- FAQ
- 文档型知识
- 流程型知识
- 快捷语/标准回复

核心实体：

- `knowledge_document`
- `knowledge_section`
- `knowledge_chunk`
- `knowledge_publish_version`

### 7.2 导入解析

支持：

- 手工录入
- PDF/Word/Markdown/HTML
- Excel 批量导入
- 第三方系统同步

### 7.3 切片入库

策略：

- FAQ 一问一答直接成 chunk
- 流程按步骤组切
- 文档按标题层级和语义块切
- 每个 chunk 只承载一个小问题

### 7.4 检索召回

链路：

- 问题标准化
- metadata 过滤
- OpenSearch 关键词召回
- OpenSearch 向量召回
- hybrid fusion
- rerank

### 7.5 运营评测

指标：

- 检索命中率
- 无答案率
- 转人工率
- 错答反馈率
- chunk 命中分布
- 过期知识比例

## 13. 扩展模块设计说明

以下模块已补入原型与交付范围，但当前属于 `Phase 1B 以后需继续细化交互` 的扩展模块。也就是说：

- 当前已完成：信息架构、页面骨架、模块边界、主要状态与后台能力占位
- 当前未完成：逐字段表单、逐操作弹窗、批量处理流、复杂表格交互、权限细粒度态

### 13.1 客服平台扩展模块

- 工单管理
  - 来源于会话转工单、留言转工单或业务流转工单
  - 关键能力：状态流转、负责人、优先级、来源会话、SLA、评论
- 留言管理
  - 关键能力：未处理/处理中/已解决、批量分配、批量改状态、导出
- 历史会话
  - 关键能力：时间/渠道/标签筛选、聊天记录查看、来源信息、回呼、导出、质检入口
- 导出管理
  - 关键能力：导出任务状态、文件下载、失败重试、来源业务识别
- 质检评分
  - 关键能力：评分维度、评分详情、复核意见、质检人信息
- CRM/业务嵌入
  - 关键能力：iframe 容器、外部标识映射、客户/订单/会员视图联动
- 会话分配规则
  - 关键能力：上次接待优先、平均分配、渠道独立路由、技能组路由
- 客服设置
  - 关键能力：在线/挂起/小休状态、自定义最大接待数、自动结束、结束前强制标签
- 视频客服
  - 关键能力：发起/接收、抓拍、静音、窗口拖拽、视频回放、摄像头默认策略
- 客服组监控 / 客服监控 / 报表中心
  - 关键能力：技能组汇总、客服个人监控、企业总览、会话报表、工作量报表、留言报表、满意度报表

### 13.2 机器人后台扩展模块

- 机器人仪表盘
  - 关注总会话量、有效会话量、问答总数、终端来源
- 机器人知识管理
  - 问题库、表格意图、流程管理、自定义寒暄库、应急关键词、知识挖掘、发布管理
- 词库
  - 相似词、停止词、实体、专有名词
- 业务设置
  - 基础设置、导航菜单、访问端设置、转人工规则、统计设置
- 数据分析
  - 热点知识、分类统计、问法统计、导航菜单统计、转人工统计、流程路径统计、留资漏斗
- 业务监控
  - 通话数、峰值 QPS、命中率、服务质量、敏感词与情绪监控
- 机器人详情 / 模拟对话
  - 机器人详情页、浏览器内测试浮层、标签化测试、点赞点踩
- 对外接口 / 参数配置
  - 问答接口、热点问题接口、逐字联想接口
  - 最低/最高阈值、挂接领域、场景、图谱、敏感词库

## 14. 风险与约束

- 当前工作区无现成代码与 git 仓库，实施阶段需按新项目结构创建
- 多端并行开发会拉高前端协作成本，需先冻结原型与接口
- AI 客服效果高度依赖知识治理与运营评测，不应过度依赖模型能力
